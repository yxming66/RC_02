# autoCtrlAPI 开发提示词

请在当前工程中开发 `autoCtrlAPI` 模块，并严格参考以下两个文件：

- `User/module/autoCtrlAPI/autoCtrlAPI开发计划表.md`
- `User/module/autoCtrlAPI/autoCtrlAPI需求说明.md`

你的任务不是一次性把所有复杂功能堆进去，而是按照计划表分阶段实现一个**可扩展、数据驱动、便于联调**的 `autoCtrlAPI` 基础框架，并在此基础上逐步补全自动控制能力。

---

## 一、开发目标
请为 `R2` 车开发下位机自动控制接口模块 `autoCtrlAPI`，用于：

- 支持区块化自动控制
- 支持基于区块转移表的模板调度
- 支持执行前 yaw 自动对正
- 支持模板化跨越动作执行
- 支持状态查询、故障处理与后续扩展

该模块定位为**局部自动执行器**，不是全场总决策器。

---

## 二、必须遵守的总体原则

### 1. 严格按计划表推进
实现顺序必须优先遵循：
1. core 定义与数学工具
2. 区块转移层
3. 查询接口
4. 调用接口层外壳
5. 基础动作层
6. step 层
7. 模板执行层
8. 基础模板
9. 故障与调试信息

不要跳过基础层直接写复杂流程。

### 2. 严格参考需求说明
所有设计都要尽量符合 `autoCtrlAPI需求说明.md` 中的约束，特别是：
- 职责边界
- 区块抽象
- yaw 零位偏置
- 模板化结构
- step 化执行
- 故障安全退出

### 3. 先搭框架，再填细节
第一版目标是：
- 架构清晰
- 接口可用
- 数据驱动
- 可以跑通最小闭环

而不是一开始就把所有模板细节写满。

### 4. 不要写成大量硬编码 if-else
不要把核心逻辑写成大量：
- `if (from == x && to == y)`

应优先使用：
- 枚举
- 结构体
- 静态表
- 查询函数
- 模板调度函数

### 5. 优先保持与现有工程风格一致
实现时请尽量匹配当前工程已有代码风格，包括：
- 文件组织方式
- 命名风格
- 头文件写法
- `C/C++` 混合工程兼容性
- 与现有 `module / device / bsp / task` 分层兼容

---

## 三、你需要完成的内容

### 第一阶段：建立基础文件与数据定义
请优先创建并实现适当的文件，例如（可根据工程风格微调）：

- `core/auto_ctrl_def.h`
- `core/auto_ctrl_math.h / .c`
- `transition/auto_ctrl_zone.h / .c`
- `transition/auto_ctrl_transition.h / .c`
- `primitive/auto_ctrl_primitive.h / .c`
- `step/auto_ctrl_step.h / .c`
- `template/auto_ctrl_template.h`
- `template/auto_ctrl_template_runner.h / .c`
- `api/auto_ctrl_api.h / .c`

需要完成：
- 核心枚举定义
- 区块属性结构体定义
- 区块转移结构体定义
- 状态、结果、故障码定义

### 第二阶段：建立静态区块表与转移表
请根据需求说明中的平台信息，先建立最小可用版本：
- `R2_ENTRY`
- `R2_EXIT`
- 平台 `1~12`

平台高度按需求说明录入。

转移表中至少先支持：
- 平台相邻转移
- 入口/出口与平台的转移
- 模板类型映射
- `required_yaw_deg`
- `yaw_tolerance_deg`
- `sensor_mode`
- `height_delta_mm`

### 第三阶段：实现查询接口
请实现：
- 区块信息查询
- 区块高度查询
- 转移表查询
- 非法转移判定

### 第四阶段：实现 autoCtrlAPI 外部接口壳子
请实现基础 API，例如：
- 初始化
- 重置
- 设置 yaw 零位 offset
- 启动一次区块转移
- 周期 update
- 查询状态
- 查询结果
- 中止

注意：
- API 层只负责对外接口与总状态机
- 不要把模板分发和基础动作直接塞在 API 文件里

### 第五阶段：实现 yaw 模块
请实现：
- yaw 归一化函数，范围 `[-180, 180]`
- 自动控制 yaw = 原始 yaw - offset
- yaw 误差计算函数
- yaw 对正判定
- yaw 对正阶段控制框架

其中：
- 角度数学建议放 `core/auto_ctrl_math.*`
- yaw 对正动作输出建议放 `primitive/auto_ctrl_primitive.*`

注意：
- `R2` 在启动区且与长方形场地边平行时读取的 yaw 作为零位偏置
- 执行区块转移前，应先校正到该转移要求的 `required_yaw_deg`

### 第六阶段：实现模板执行框架
请先实现 step 层，再实现模板层最小调度器，而不是一开始写满所有复杂细节。

建议分解为：
- `step` 层：step 上下文、step 执行、step 结果
- `template` 层：按模板编号调度 step 序列

至少要具备：
- `IDLE`
- `PREALIGN`
- `RUN_TEMPLATE`
- `SUCCESS`
- `FAIL`
- `ABORT`

模板内部必须以 step 机制组织，第一版可以是最小 step 集。

### 第七阶段：先实现最小模板集
优先级顺序如下：
1. `FLAT_MOVE`
2. `ASCEND_200_STD`
3. `DESCEND_200_STD`
4. `ASCEND_200_NO_FRONT_SICK`
5. `DESCEND_200_NO_FRONT_SICK`
6. `ASCEND_400_STD`
7. `DESCEND_400_STD`

如果第一轮实现复杂度过高，可以先把后几类模板留作占位接口，但整体框架要留好。

## 四、关键设计约束

### 1. yaw 处理要求
必须使用“零位偏置”方案：
- 不是把上电原始 yaw 直接当绝对角
- 而是读取一次零位 offset 后统一换算

### 2. 下台阶逻辑要求
下台阶应视为上台阶过程的逆过程。
当前理解下：
- 下台阶通常保持上台阶完成后的朝向不变
- 不要求在全局状态机里维护复杂 heading 状态
- 但转移项必须给出执行前所需 yaw

### 3. 对正逻辑要求
在模板执行前必须先做 yaw 对正。
未对正成功，不得进入模板执行。

### 4. 特殊平台要求
对于 `6`、`8` 等前向 `SICK` 不适合直接判定的场景，必须为后续保留：
- 特殊模板
- 特殊传感器模式
- 光电主导判据

### 5. 故障处理要求
每个关键阶段都应考虑：
- 超时
- 失败
- 安全退出

即使第一版只写框架，也必须预留故障码和失败出口。

---

## 五、实现时的具体要求

### 1. 尽量先复用现有工程能力
如果工程中已有以下能力，请优先复用而不是重复造轮子：
- IMU/yaw 获取接口
- `SICK` 距离读取接口
- 光电读取接口
- pole 控制接口
- chassis 控制接口

并建议把这些外部能力的接入点放在基础动作层或其适配层，不要散落在模板层与 API 层。

### 2. 暂时不确定的外部依赖，可先做适配层/占位接口
如果某些底层接口尚不明确：
- 可以先声明适配函数
- 可以先写 TODO 注释
- 但不要阻碍整体框架建立

### 3. 代码必须有清晰注释
特别需要注释清楚：
- 区块定义依据
- 平台高度依据
- yaw 零位含义
- 转移表字段含义
- 状态机流程
- 模板占位与后续扩展点

### 4. 注意可编译性与最小闭环
请优先保证：
- 头源文件结构正确
- 依赖关系清晰
- 基础代码可以编译
- 最小状态机闭环成立

同时优先保证分层边界清晰：
- primitive 不含模板分发
- template 不含区块表
- api 不含 step 细节

不要为了“写全”而让工程结构变乱。

## 六、建议的输出风格
请在开发过程中优先产出：
1. 清晰的数据结构
2. 清晰的 API
3. 清晰的状态机
4. 清晰的转移表
5. 必要注释

如需分步提交，优先按以下顺序完成：
- 第一步：core 头文件、枚举、结构体、角度数学
- 第二步：区块表、转移表、查询函数
- 第三步：API 骨架
- 第四步：primitive 原语
- 第五步：step 框架
- 第六步：template 框架
- 第七步：最小模板实现

## 七、交付要求
请最终至少做到：
- 工程中存在完整的 `autoCtrlAPI` 基础文件结构
- 区块表与转移表已落地
- `autoCtrlAPI` 基础接口已落地
- yaw 零位与 yaw 对正基础逻辑已落地
- 模板调度状态机已落地
- 至少一个最小模板链路可跑通
- 为后续继续扩展模板和传感器联调预留好接口

如果某些底层控制接口尚未明确，请用占位实现和注释明确说明，不要擅自假设过多不可验证细节。
